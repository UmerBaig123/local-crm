Local-First Desktop CRM

Two paths to building a keyboard-first, privacy-focused CRM for solo consultants. Same destination, different risk profiles.

May 22, 2026 — Decision Document

Decisions That Don't Change

These are the same regardless of which path you choose.

Database

SQLite + FTS5

CRM is transactional (CRUD), not analytical. SQLite is the only serious option for embedded local-first. FTS5 gives sub-50ms full-text search across 100K+ contacts. Portable single-file database. No alternatives come close for this workload.

Target Audience

Solo Consultants ($100K+ revenue)

4.7M+ in the US. Clear pain points: privacy (HIPAA, client confidentiality), offline access, speed. Willing to pay $50-100/month. No desktop-native CRM competitor exists for this segment today.

Custom Fields

Hybrid: Built-in columns + JSONB

Core fields (name, email, phone, company) as indexed columns for fast queries. Everything else in a JSON blob. 80/20 optimization: fast queries on common fields, full flexibility for custom fields. Avoids EAV performance problems.

Keyboard Architecture

cmdk + Global Shortcuts + react-focus-lock

Command palette via cmdk (used by Linear, Raycast). Gmail-style navigation (j/k, g+c). Context-aware actions. react-focus-lock for modal focus trapping and accessibility. This is the core differentiator.

Virtual Scrolling

@tanstack/react-virtual

Handles 1M cells at 60fps. Variable row heights (contacts with notes are taller). Modern hooks-based API. react-window is legacy, react-virtualized is deprecated. Only real option in 2026.

Sync Strategy (Phase 2)

PowerSync (deferred to Month 2-3)

Both paths ship local-only first. Sync adds conflict resolution, offline queues, merge logic. Doubles MVP complexity. Add PowerSync ($49/month, Postgres backend) after core CRM is validated. Turso is a backup option.

Path A picks Electron because it eliminates the Rust learning curve. You already know the web stack. Superhuman proved Electron can be optimized to feel fast. The 3.2s cold start is still 2x faster than Salesforce. Larger ecosystem means fewer roadblocks. Development speed is the highest of any option: full npm access, instant hot reload on both frontend and backend, 13 years of Stack Overflow answers for every edge case. When you hit a wall at 2am, Electron has the answer already written.

Path B picks Tauri because the core value proposition is SPEED. 1.4s vs 3.2s is noticeable. 8 MB vs 165 MB signals quality. If you're charging premium ($75/month), the product should feel premium. Rust learning cost (1-2 weeks) pays for itself in performance. Tauri is younger (v2 released 2023) and has fewer production apps to point to, but the ones that exist (Windsurf, Crabnebula) report stability. The ecosystem is growing fast (55% YoY GitHub growth), and for a CRM the backend is thin: SQLite reads/writes and file operations. You won't need deep Rust expertise.

The dev speed gap is real but narrowing. In the first 2 weeks with Tauri, expect ~60-75% productivity vs Electron (Rust compiler fights, unfamiliar error messages, fewer examples). By week 3-4, productivity normalizes to ~85-90% as the backend stabilizes and most work shifts to the React frontend, which is identical in both frameworks.

The maturity gap matters most for edge cases. Tauri uses the OS native webview, which means CSS rendering can differ slightly between Mac (WebKit), Windows (WebView2/Chromium), and Linux (WebKitGTK). Electron ships its own Chromium, so rendering is identical everywhere. For a CRM (tables, forms, lists), this is unlikely to be a problem. For pixel-perfect design or complex animations, it could bite you.

Key question: Is the 56% speed improvement (1.4s vs 3.2s) worth slower development velocity for 2-3 weeks and a less battle-tested platform? Both are faster than every web CRM. The difference is whether "fast" or "fastest" matters more, and whether you value shipping speed or runtime speed.

Not controversial. CRM is a transactional workload: create contacts, update contacts, search contacts. SQLite handles 10,000+ reads/second, 5,000+ writes/second in WAL mode. FTS5 provides full-text search with porter stemming and trigram support under 50ms for 100K contacts. Single-file database makes backup trivial. Every alternative has a fundamental mismatch: DuckDB is for analytics, Realm is mobile-first with MongoDB lock-in, RocksDB has no SQL. SQLite is the obvious and correct choice.

Timeline & Buffer Path A: 4 weeks Path B: 3 weeks

3x parallel Claude Code instances compress timeline dramatically
▶
Development model: 3 parallel Claude Code instances. You're not writing code line-by-line. You're orchestrating 3 AI agents simultaneously, each working on independent modules. This compresses parallelizable work by ~3x. Sequential work (foundation, integration, final polish) remains 1x. The bottleneck shifts from coding speed to architectural decisions and integration testing.

Why the compression works: After foundation is set up (sequential, ~3-5 days), the codebase splits into independent modules. Instance 1 builds tags + custom fields. Instance 2 builds CSV import/export. Instance 3 builds keyboard shortcuts + command palette. In Path A, one instance also handles the deals pipeline in parallel. These modules touch different files and don't conflict. Integration at the end (merging, testing interactions between modules, polish) is the remaining sequential bottleneck.

Where it doesn't compress: Architectural decisions (schema design, component structure, state management) still need your brain. Foundation/scaffolding must be done first so instances have a shared base. Integration testing at the end can't be parallelized. The real timeline risk shifts from "coding speed" to "merge conflicts and integration bugs" between parallel streams.

Realistic constraint: 3 instances produce code fast but can make inconsistent decisions (different naming conventions, state patterns, error handling). Budget 2-3 days at the end for unifying code style, fixing integration issues, and ensuring the modules work together. This is where your judgment matters most.

Risks

Threats to both paths and how to handle them.

Rust Learning Curve (Path B only)

MEDIUM — Path B only

Tauri backend requires Rust. The CRM backend is simple CRUD, but Rust's ownership model and compiler can slow you down significantly in week 1-2.

Mitigation: The Tauri backend is thin (SQLite commands, file operations). Most complexity lives in the React frontend. If Rust blocks you for more than 3 days, switch to Electron. You'll lose 3 days, not 3 weeks.

Contacts-Only Too Minimal (Path B only)

HIGH — Path B only

Solo consultants already have contacts in Google Contacts, Apple Contacts, Notion. Paying $75/month for another contact manager with keyboard shortcuts is a hard sell without deals/pipeline.

Mitigation: Position as "power user contact manager" not "CRM." Target people who hate their current contact management, not people looking for a CRM. Alternatively, add a simple deal stage tag instead of a full pipeline (1 day work).

No Sync = No Adoption

MEDIUM — Both paths

Consultants work on multiple devices. A CRM that lives on one machine is friction. They'll use it for a week, switch devices, and stop.

Mitigation: Add Litestream backup in Week 4-5 (1 day work). Users can backup to S3 and restore on another machine. Not real-time, but functional. Prioritize PowerSync integration immediately after MVP.

Desktop App Distribution

MEDIUM — Both paths

Code signing certificates ($99/year Apple, $200-400 Windows), auto-update infrastructure, and OS-specific bugs. Users may not trust downloading a .exe/.dmg from a new company.

Mitigation: Budget 1 week for installers and signing. Use Tauri's built-in updater or electron-updater. Launch on Mac first (consultants skew Mac), add Windows later. Consider Mac App Store for trust.

Market Doesn't Want Desktop

MEDIUM — Both paths

The trend is toward web apps, not desktop. Consultants might prefer accessing their CRM from any browser on any device. "Local-first" might be a feature for developers, not consultants.

Mitigation: This is the core hypothesis. Validation (either landing page or alpha testing) answers this in week 2-4 before full investment. If consultants say "I need web access," pivot to a web app with local caching instead.

Solo Founder Burnout

MEDIUM — Both paths

3-4 weeks of intense development orchestrating parallel AI instances with no revenue and uncertain market validation. Pace is fast but can be mentally taxing.

Mitigation: Set a hard kill date. If no paying customer by Week 6, stop and reassess. Build in public (Twitter/LinkedIn) for external accountability. The compressed timeline actually helps: you get to validation faster, reducing the uncertainty window.

The Bottom Line

Both paths lead to the same product. A local-first, keyboard-driven desktop CRM for solo consultants. The difference is sequencing: Path A ships a more complete product slower with less risk. Path B ships a thinner product faster with more risk.

If you want to minimize risk: Path A. Electron, contacts + deals, $50/month, 4 weeks. You'll ship a product that competes with Folk and Attio on features while being faster. Validate with a real alpha, not email signups.

If you want to move fast and bet on speed as the differentiator: Path B. Tauri, contacts-only, $75/month, 3 weeks. You're betting that "fastest CRM on earth" is compelling enough without a deals pipeline. Higher upside if it works, higher risk if consultants want a full CRM.

What I'd actually do: Take Path A's technology choice (Electron) with Path B's scope (contacts-only) and Path A's validation (build alpha, show people). Ship a fast Electron contacts app in ~2.5 weeks with 3 parallel instances, show it to 20 consultants, and let their feedback tell you whether to add deals next or whether the whole thesis is wrong. Price at $50/month to start, raise to $75 after adding deals + sync. With this dev velocity, you can have deals shipped within a week of deciding to add them.

The one decision that matters most: Validation timing. Whether you build alpha first or run ads first determines everything. But with 3 parallel Claude Code instances, the build is so fast that "build alpha then validate" is now strictly better than "run ads for 30 days." You can have a working product in people's hands before a landing page ad campaign would even finish running.
